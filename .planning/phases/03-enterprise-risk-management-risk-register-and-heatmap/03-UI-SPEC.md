## Spacing Scale

Unchanged from Phase 1 (source: 01-UI-SPEC.md):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge padding, inline element gaps |
| sm | 8px | Input internal padding (vertical), label-to-field gap |
| md | 16px | Default element spacing, card internal padding, form field gap |
| lg | 24px | Card-to-card gap, section padding, form section gap |
| xl | 32px | Page content top padding, layout column gap |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level breathing room |

Phase 3 exceptions:
- Heatmap grid cell: 64px × 64px fixed (not a spacing token — grid geometry)
- Heatmap cell gap: 2px (sub-token, intentional — tight grid conveys matrix density)
- Risk dot: 24px × 24px minimum touch target within heatmap cell
- Filter bar inputs: 44px height (WCAG touch target minimum — same as Phase 1 rule)
- 5×5 matrix selector cells (interactive scoring): 48px × 48px each (WCAG 2.5.5 compliant)

**60/30/10 split:**
- 60% dominant: `paper` (#F3F7FD) — page background, heatmap page background
- 30% secondary: white (#FFFFFF) — cards, register panel, heatmap card, filter bar card
- 10% accent: `gold` (#C8A44A) — reserved elements listed below

**Accent (#C8A44A) is reserved exclusively for (Phase 3 additions):**
- "Add Risk" primary CTA button
- "Save Risk" / "Save Treatment" submit buttons
- Active navigation item indicator (inherited from Phase 1)
- Focus ring on all interactive elements (2px solid gold, 2px offset — inherited)
- Selected cell highlight on the 5×5 matrix selector (gold border, 2px solid)

**Severity color system (Phase 3 specific — uses existing tokens):**

| Severity | Score Range | Background Class | Badge Class | Dot Background |
|----------|------------|------------------|-------------|----------------|
| Low | 1–4 | `bg-ok/15` | `bg-ok/10 text-ok border-ok/30` | `bg-ok` |
| Medium | 5–9 | `bg-warn/20` | `bg-warn/10 text-warn border-warn/30` | `bg-warn` |
| High | 10–15 | `bg-err/20` | `bg-err/10 text-err border-err/30` | `bg-err` |
| Critical | 16–25 | `bg-err/40` | `bg-err/30 text-err border-err/50` | `bg-err` with ring |

Distinguishing High from Critical: High uses `bg-err/20`, Critical uses `bg-err/40` — same hue, stronger opacity.
Critical risk dots additionally have a 2px white ring `ring-2 ring-white ring-offset-1` to convey elevated urgency.

**Treatment status colors (Phase 3 specific):**

| Status | Badge Class |
|--------|------------|
| planned | `bg-paper text-navy-mid border-paper-border` |

**Contrast requirements (new combinations):**
- `ok` (#27AE60) on `ok/10` tint: navy text (#0B1625) used in badge — passes WCAG AA
- `warn` (#E67E22) on `warn/10` tint: passes WCAG AA at 4.6:1
- `err` (#E74C3C) on `err/10` tint: passes WCAG AA at 4.5:1
- White text on `ok` (#27AE60) for risk dots: 3.1:1 — acceptable for non-text indicator use

### 23. 5×5 Matrix Selector (Interactive Scoring)

**Custom component** — built on `button` HTML elements, styled with Tailwind
**Used on:** Risk create/edit form (`/risk/new`, `/risk/[id]/edit`) — used twice (inherent + residual)
**File:** `components/risk/MatrixSelector.tsx`

Layout: 5×5 grid of 25 clickable cells, severity-colored, with axis labels.

```
[MATRIX CONTAINER — inline-block]
  [Y-AXIS LABEL — "Likelihood" — DM Sans 10px/500 navy-mid uppercase, vertical text left side]

  [FLEX ROW — gap-4px]
    [Y-AXIS NUMBERS — flex column, justify-between, gap-4px]
      [5, 4, 3, 2, 1 — DM Sans 11px/400 navy-mid, width 12px text-right]

    [GRID — CSS grid, 5 cols × 5 rows, gap-4px]
      [For each of 25 cells — <button> element]
        [Default: severity bg color at 15-40% opacity, border transparent, 48px×48px, rounded-[4px]]
        [Hover: severity bg at 40-70% opacity, border 1px paper-border, transition 120ms]
        [Selected: severity bg at 80-100% opacity, border 2px solid gold, ring-2 ring-gold/30]
        [Focus: border 2px solid gold, gold ring offset 2px]
        [Content: empty (color communicates zone) — aria-label="Likelihood [L], Impact [I], Score [score], [Severity]"]

  [X-AXIS NUMBERS — 1 2 3 4 5 — DM Sans 11px/400 navy-mid, centered below each column]
  [X-AXIS LABEL — "Impact" — DM Sans 10px/500 navy-mid uppercase, centered]

  [SELECTION DISPLAY — mt-8px, flex row, gap-8px, align-items center]
    [If selected: "L[N] × I[N] = [score]" — DM Mono 13px/500 navy-900]
    [Severity badge for current selection]
    [If not selected: "Select a cell to score" — DM Sans 13px/400 navy-mid italic]
```

Grid orientation convention (CRITICAL — matches heatmap, source: 03-RESEARCH.md Pitfall 7):
- Row 0 (top) = Likelihood 5; Row 4 (bottom) = Likelihood 1
- Col 0 (left) = Impact 1; Col 4 (right) = Impact 5
- High-likelihood × high-impact (Critical zone) is TOP-RIGHT — standard ERM presentation

Interaction rules:
- Click a cell to select — only one cell selectable per instance (single selection)
- Keyboard: Arrow keys move selection within grid; Enter/Space confirms
- Tab moves focus into grid; first cell in grid is focus origin
- Selected cell: `aria-pressed="true"` — all others `aria-pressed="false"`

Used twice on the Create/Edit Risk form:
- Instance 1: Inherent risk scoring (always required — labeled "Inherent Risk Score")
- Instance 2: Residual risk scoring (optional — labeled "Residual Risk Score", rendered below mitigating controls field)

The two instances are visually separated by a section divider (`1px paper-border, my-24px`) and distinct section headings.

### 28. Inline Treatment Status Select

**shadcn primitive:** `Select`, `SelectTrigger`, `SelectContent`, `SelectItem` (already installed)
**Used on:** Risk detail page (`/risk/[id]`) — each treatment row, right side of Row 1
**File:** `components/risk/TreatmentStatusSelect.tsx`

Pattern: Controlled `<Select>` rendered in place of the static Treatment Status Badge. On value change, calls a Server Action (`updateTreatmentStatus`) and fires a success toast. No page reload.

```tsx
<Select value={treatment.status} onValueChange={(val) => handleStatusUpdate(treatment.id, val)}>
  <SelectTrigger className="h-7 w-[130px] text-[13px] border-paper-border rounded-grc-sm">
    <SelectValue />
  </SelectTrigger>
  <SelectContent>
    <SelectItem value="planned">Planned</SelectItem>
    <SelectItem value="in_progress">In Progress</SelectItem>
    <SelectItem value="completed">Completed</SelectItem>
    <SelectItem value="cancelled">Cancelled</SelectItem>
  </SelectContent>
</Select>
```

Style notes:
- Trigger height: `h-7` (28px) — compact, fits Row 1 without increasing row height
- Trigger width: `w-[130px]` — matches approximate width of longest status label "In Progress"
- Trigger border: `border-paper-border` — low-visual-weight; does not compete with the title
- Font: `text-[13px]` (DM Sans via shadcn default) — matches Treatment Status Badge text size
- On hover: `border-navy-mid/40` — subtle affordance that the element is interactive
- "Overdue" is NOT an option in this select — it is set automatically by the system; if current status is `overdue`, the trigger displays a read-only Treatment Status Badge (Component 21) instead of an editable Select

Overdue guard: if `treatment.status === 'overdue'`, render the static `TreatmentStatusBadge` (Component 21) instead of this component. The overdue state is system-set — users cannot manually select it.

Interaction:
- `onValueChange` fires immediately on selection (no confirm step for non-destructive status changes)
- Changing to `cancelled` triggers the existing "Cancel treatment?" Dialog (Destructive Confirmation, inherited from Phase 1) before committing
- Server Action: `updateTreatmentStatus(treatmentId: string, newStatus: TreatmentStatus)` — validates transition, records in audit log, checks overdue, sends Resend notification if overdue triggered
- Success: toast "Treatment status updated." (4000ms success)
- Failure: toast "Unable to save. Please try again." (persistent destructive)
- Optimistic update: trigger shows new value immediately; reverts on server error

ARIA: `<SelectTrigger aria-label="Update treatment status for [treatment title]">`
